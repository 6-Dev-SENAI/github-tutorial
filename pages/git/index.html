Git

O que é o Git?
O Git é um sistema de controle de versão open-source, ou seja, ele é gratuito. Ele é utilizado para a criação de um histórico de alterações em código-fonte de projetos de desenvolvimento de software. Foi desenvolvido por Linus Torvalds, o criador do sistema operacional Linux. O Git é a melhor escolha para a maioria das equipes de software. Por meio de sua utilização, podemos saber quais foram as alterações realizadas, quem fez cada uma das alterações e baixar essas mudanças em nossa máquina. Se necessário, revertê-las para uma versão anterior. Com isso nos ajuda muito a controlar o fluxo de novas funcionalidades entre vários desenvolvedores no mesmo projeto com ferramentas para análise e resolução de conflitos quando o mesmo arquivo é editado por mais de uma pessoa em funcionalidades diferentes.
A importancia da Utilização do Git
• Trabalho de Equipe: Você trabalha com mais duas pessoas da área de programação em um projeto, fez algumas alterações no código na sua máquina, criando uma funcionalidade para o sistema, e quer compartilhá-la com seus colegas. Como você faria isso?
• Dados mais seguros: Você trabalhou durante duas semanas em uma função nova para o programa e, quando chega ao trabalho, sua máquina sofre uma pane e apaga o disco. Como você recupera seu trabalho?
• Gestão de mudanças: Você fez algumas alterações no código e elas foram publicadas, porém, o sistema teve um comportamento estranho e acabou apresentando erros. Como você retorna para a versão anterior? Com o Git isso é possível

Como configurar o Git Localmente

Para configurar o git localmente, será necessário ter a sua conta no github!
Para prosseguir, abra o terminal de comando (o CMD) e escreva, com o git já instalado na sua máquina:

$ git config --global user.name "< seu_nome_no_git_hub >"

Pressione < Enter >, assim você já define o seu usuário no git local e já precisamos configurar o e-mail

$ git config --global user.mail "< seu_email_no_git_hub >"

Pressione < Enter >, assim você já configurou seu e-mail do git localmente!

Pronto, agora o grande segredo vem aqui:
A próxima vez que der git push em algum repositório, ele pedirá a sua senha ou terá que configurar a sua senha traves do token de github
Ambas as maneiras são simples, basta seguir o passo a passo que o próprio git mostrará!

Comandos
$ git clone < link > | $ git init -> $ git remote add origin < link_clone >

Com esse comando você dá um clone no seu repositório online, fazendo assim com que todos os arquivos que estão lá venham para sua máquina

Caso prefira usar o git init, mais pra frente você precisará usar o git remote add origin < link_clone >
O git init inicia um repositório localmente, na sua máquina, para então subir à um repositório online!
Para a próxima etapa, será necessário criar um repositório no GitHub, para pegar o link do repositório remoto

Assim que criado, siga os procedimentos de

$ git add remote origin < link_clone >
onde o < link_clone > é o link do repositório criado para subir os códigos remotamente.

$ git add .

Assim que feito o

$ git clone | $ git init
Você estará aberto para desenvolver o que for necessário dentro dos seu reposítório local!

Quando finalizar, adicione todos esses aquivos novos ou modificados ao seu repositório remoto/local
O repositório remoto/local é um ambiente em que seu repositório local se "conecta" ao repositório remoto, antes de qualquer tipo de commit ou push.

Para adicionar os arquivos, use o

$ git add .
E pronto, agora vamos às próximas etapas.

$ git commit -m "Mensagem exemplo"

Para a confirmação da adição dos arquivos ao seu repositório remoto/local, faça um "commit"
Com o commit, você dará uma mensagem auto-explicativa de todo o trabalho feito até aquele momento.

$ git commit -m "Até o momento, estamos confirmando os arquivos ao repositório remoto/local"
O "-m" em conjunto com as aspas duplas determina que você quer colocar uma mensagem no seu commit, sempre é recomendado usar, pois serve como uma documentação dos arquivos desenvolvidos até agora.

Finalmente, chegamos ao momento de enviar ao repositório online!

$ git push
Com esse comando, empurramos os nossos arquivos ao repositório online tudo aquilo que adicionamos com o
$ git add .
$ git commit -m "Mensagem exemplo"
Agora, assim que abrirmos o nosso repositório online, já estará tudo salvo lá com segurança!

$ git pull

Imaginando que já estamos com o repositório pronto, feito por algum colega, podemos puxar esses arquivos do repositório online através do comando acima.

Quando executamos o comando acima, uma cópia do repositório na nossa máquina criando assim um modelo remoto/local onde trabalharemos e seguiremos aquele ritmo de:

$ git add .
$ git commit -m "Mensagem exemplo"
$ git push

$ git Branch
$ git branch < nome_branch >

Ao abrirmos uma nova branch, criamos uma nova cópia da master. A partir de um ponto da história determinamos o início de um novo caminho, chamado de branch.

Nesse caminho, somos livres a seguir com o desenvolvimento, tendo em mente de que precisamos sempre manter esse novo caminho alinhado com a master, nosso caminho principal.

$ git checkout < nome_branch >

O checkout é o modo de indicarmos ao git que queremos ir pra um outro caminho, independente de qual seja.
Para o exemplo a seguir, você se encontra na master * e quer seguir para uma branch chamada INDEX:

$ git checkout INDEX
Fazendo isso, saímos do caminho da master e seguimos para o caminho da INDEX (independente do que esteja sendo desenvolvido lá).

Caso esteja trabalhando com um grupo de desenvolvedores, é possível que queira conferir uma branch, um caminho, de algum de seus colegas de equipe.
Para isso, precisa passar para um caminho REMOTO, ou seja, ele não se firma na sua máquina de maneira local, porém você tem acesso e consegue alterar as informações contidas nele.
Exemplo, você está no caminho INDEX que está local na sua máquina e deseja passar pra um caminho REMOTO, chamado de FEATURE, de um dos seus colegas.

$ git checkout REMOTES/ORIGIN/FEATURE
E pronto, o seu acesso será igual remota ou localmente.

$ git push --set-upstream origin < nome_branch >

A primeira vez que você cria uma branch, você segue com o caminho normal do add e commit, e no momento do push que muda. É necessário adicionar esse branch dentro do repositório online do git.
Vamos adicionar a branch que criamos anteriormente, chamada INDEX, ao nosso repositório online :

$ git push --set-upstream origin INDEX
Lembrando que para usarmos o push, segue as mesmas regras de add e commit :
$ git add .
$ git commit -m "Mensagem exemplo"

$ git branch -a

Esse comando é para listagem de todas as branchs que seu repositório possue, sejam elas locais ou remotas.

$ git branch -a

// Saída

* INDEX // branch selecionada
master // branch local
remotes/origin/master // branch remota
remotes/origin/FEATURE // branch remota
$ git merge < branch >

Com o merge nós unimos tudo aquilo que foi feito na nossa branch, seja remota/local, à nossa master. Tomando sempre o cuidado para que não haja conflito entre as linhas de código.
Caso aconteça um conflito, é necessário arrumar essas linhas de código conflitadas na mão, alterando até que o git reconheça que está ok.

Para começarmos a sequência mais delicada das branchs, vamos mudar primeiro pra a master, onde iremos fazer nosso merge.

$ git checkout master
Aqui na master, vamos puxar e unir (merge) os códigos da nossa branch INDEX à master.
$ git merge INDEX

Caso não haja conflitos entre os arquivos, os merge será realizado e assim a única coisa que vamos ter que fazer é o push para oficializar, de maneira remota/online o nosso merge entre as branchs.

$ git push
Feito isso, oficializamos o nosso merge e podemos sair e eliminar essa branch.

Após terminarmos os desenvolvimentos na nossa branch, o que devemos fazer?

Muitos dizem que devemos manter as nossas branchs, para entendermos o que fizemos durante aquele caminho, naquele momento da história. Mas realmente precisamos manter?
Sim e não; só depende do seu time de devs ou do seu Tech Lead, de qualquer forma, mostraremos como deletar essa branch, de uma vez, de maneira remota e local, com o comando:

$ git push origin --delete < nome_branch >

Fazendo isso, é excluído do momento da história, aquela branch no local e remoto.
Seguinda a ideia dos exemplos acima, já fizemos o merge na branch INDEX; agora podemos deletar essa branch do nosso repositório.

$ git push origin --delete INDEX
Impossibilitando a recuperação desses arquivos pela branch, porém já temos acesso aos mesmos pela master.

Finalmente
Com todos esses comandos, já conseguimos copiar um repositório online pra nossa máquina local; podemos alterar e adicionar arquivos no repositório local;
Logo depois disso, já adicionamos, "commitamos" e subimos (com o push) ao repositório online; assim como aprendemos a usar as branchs, caminhos diferentes na história, e o merge para unir cada branch à nossa branch principal, que é sempre a master.
